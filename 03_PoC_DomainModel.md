# ドメインモデル：claude-discord-bridge PoC

**スコープ: PoC（概念実証）範囲のみ**

---

## 1. ドメインモデル概要

本ドキュメントは、PoC におけるシステムの「何をやるか」をドメインモデルとして定義する。

### 1.1 PoC で実現すること

- Discord のテキストチャンネルを通じて、ClaudeCode と対話する
- メッセージを送るだけで自動的にセッションが開始・継続される
- 処理の中断やセッションのリセットをコマンドで制御する
- 許可されたユーザーのみが操作できる
- ClaudeCode の出力を Discord の制約に合わせて分割・送信する

### 1.2 PoC で実現しないこと

- フォーラムチャンネル / スレッド管理
- 複数セッション / 並列実行
- 会話フォーク（`!history` / `!fork`）
- フォルダ選択 UI（ドリルダウン）
- 過去セッションの呼び出し（`!resume` / `!sessions`）

## 2. ドメインオブジェクト

### 2.1 Session（セッション）

システムの中核概念。ClaudeCode との一連の対話を表す。
PoC では同時に 1 つのセッションのみ存在する。

**属性：**

| 属性 | 説明 |
|------|------|
| セッション ID | ClaudeCode の `--session-id` に渡す識別子。会話の継続性を保証する |
| 状態（State） | セッションの現在の状態（後述の状態遷移を参照） |
| 作業ディレクトリ | ClaudeCode が操作するディレクトリ（`.env` の `WORK_DIR` で固定） |

**セッションの自動管理：**

- **自動開始**: 初回のユーザー入力時にセッション ID が自動生成される。明示的な開始コマンドは不要
- **自動継続**: 同じセッション ID を `--session-id` に渡すことで、会話が自動的に継続する
- **明示的リセット**: `!new` コマンドでセッション ID をリセットし、新しい会話を開始する

**状態遷移：**

サブプロセス方式では、状態は **claude プロセスの有無** で決まる。

```
Idle（待機中）◄──────────┐
    │                      │
    │  ユーザー入力         │  プロセス終了（正常/異常）
    │  → claude -p 起動    │
    ▼                      │
Busy（処理中）────────────┘
    │
    │  !interrupt → SIGINT 送信 → プロセス終了
    ▼
Idle（待機中）
```

プロセスの生存 = Busy、不在 = Idle であり、**状態の不整合が原理的に起きない**。

**状態ごとの振る舞い：**

| 状態 | ユーザー入力時の振る舞い | コマンドの受付 |
|------|--------------------------|----------------|
| Idle | セッション ID がなければ生成し、`claude -p` を起動。Busy に遷移 | `!new` |
| Busy | 「処理中です」と返し、入力を拒否 | `!interrupt`, `!new` |

### 2.2 Command（コマンド）

Discord から届いたテキストを分類する概念。すべてのユーザー入力は、以下のいずれかに判定される。

| 種別 | トリガー | 説明 |
|------|----------|------|
| NewCommand | `!new` | セッション ID をリセットし、新しい会話を開始する |
| InterruptCommand | `!interrupt` | 実行中のプロセスを中断する（SIGINT 送信） |
| PromptInput | 上記以外のテキスト | ClaudeCode への入力として転送する |

**コマンドの有効性ルール：**

| コマンド | Idle | Busy |
|----------|------|------|
| `!new` | ✅ セッション ID をリセット | ✅ プロセスを中断してからリセット |
| `!interrupt` | ❌ 無効（処理なし） | ✅ 実行 |
| PromptInput | ✅ claude -p 起動 | ❌ 処理中 |

### 2.3 AccessControl（アクセス制御）

メッセージ処理の最初の関門。すべてのメッセージに対し、以下の条件を満たすかを判定する。

| 条件 | 説明 |
|------|------|
| ユーザー ID | `.env` の `ALLOWED_USER_IDS` に含まれるか |
| チャンネル ID | `.env` の `CHANNEL_ID` と一致するか |

条件を満たさないメッセージは、一切の応答なく無視する（存在しないものとして扱う）。

### 2.4 BotResponse（Bot 応答）

ClaudeCode の出力またはシステムメッセージを Discord に送信する責務を持つ。

**種別：**

| 種別 | 内容 |
|------|------|
| ClaudeCodeOutput | ClaudeCode からの応答テキスト |
| SystemMessage | セッションリセット / エラーなどのシステム通知 |
| RejectionMessage | 処理中の入力拒否などの通知 |

**分割ルール：**

Discord のメッセージ上限（2000文字）を超える場合、複数メッセージに分割して送信する。

## 3. ドメインルール

### 3.1 セッションの自動開始

ユーザーが初めてメッセージを送信した時点で、セッション ID が自動生成される。明示的な開始コマンドは不要。

### 3.2 処理中の入力拒否

セッションが Busy 状態のとき、PromptInput は拒否する。ただし `!interrupt` と `!new` は受け付ける。

### 3.3 アクセス制御の優先

アクセス制御の判定は、コマンド解析やセッション状態の確認よりも先に行う。許可されないユーザーからのメッセージは、コマンドの種別に関わらず無視する。

### 3.4 プロセス異常終了時の挙動

`claude -p` プロセスが異常終了（非ゼロの exit code）した場合、エラーメッセージを Discord に送信し、セッションは Idle 状態に戻る。セッション ID は維持されるため、次のメッセージで会話を継続できる。

---

## 4. メッセージ処理フロー

ユーザーからのメッセージが処理される流れを以下に示す。

```
Discord メッセージ受信
    │
    ▼
[AccessControl] ── 拒否 → 無視（応答なし）
    │ 許可
    ▼
[Command判定] ── コマンド種別を特定
    │
    ├─ NewCommand ──────→ Busy 中ならプロセス中断 → セッション ID リセット → 「新しいセッションを開始」応答
    ├─ InterruptCommand → Busy なら SIGINT 送信 → 「中断しました」応答 / Idle なら無視
    └─ PromptInput ────→ [Session 状態チェック]
                              ├─ Idle → セッション ID なければ生成 → claude -p 起動 → Busy
                              └─ Busy → 「処理中です」応答（入力拒否）
```

## 5. 環境変数（PoC）

| 環境変数 | 説明 |
|----------|------|
| `DISCORD_TOKEN` | Discord Bot の認証トークン |
| `CHANNEL_ID` | Bot が動作するテキストチャンネルの ID |
| `ALLOWED_USER_IDS` | 操作を許可する Discord ユーザー ID（カンマ区切り） |
| `WORK_DIR` | ClaudeCode の作業ディレクトリ（固定値） |
